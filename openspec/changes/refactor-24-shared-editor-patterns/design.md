## Context

`TaskBottomSheetContent` and `HabitBottomSheetContent` (and their `*Models.kt`/`*FormSections.kt`/`*FooterSection.kt` siblings) independently implement the same editor mechanics: snapshot-based dirty tracking, debounced autosave with flush-on-dismiss, discard-confirmation wiring through `TempoModalBottomSheet`, a Cancel/Confirm/Delete footer, property-row field layout, and two-layer (ViewModel + use case) title/description validation. Both were built to the same shape independently (`TaskBottomSheetModels.kt` even comments that it "mirrors the `HabitFormSnapshot` pattern"), so the duplication is structural, not superficial. `presentation-architecture` already codifies a guardrail from the #753 refactor that bottom-sheet god files be decomposed into focused units while keeping `TaskBottomSheet`/`HabitBottomSheet` entry points stable — this change extends that guardrail to also require the *shared* mechanics be implemented once.

Constraints: domain code (use cases, `ValidationUtils`) must stay pure Kotlin with no `android.*`/Compose imports. UI code keeps the Screen/Content split. Public entry composables `TaskBottomSheet` / `HabitBottomSheet` and the `TasksViewModel` / `RoutinesViewModel` event APIs must not change signature. No persisted data model, navigation, or user-visible behavior changes — this is purely a duplication-reduction refactor.

## Goals / Non-Goals

**Goals:**
- One implementation each of: snapshot/dirty-diff + debounced autosave + flush-on-dismiss, the Cancel/Confirm/Delete footer, property-row field layout, title/description `TextField` styling, the `rememberSaveable` editor-key derivation, shared constants, `toUserFacingMessage()`, and title/description validation — reused by both Tasks and Routines.
- Preserve all feature-specific behavior unchanged: task periodicity/scheduling UI, habit Build/Quit switching, habit-chain member selection and its confirmation dialogs, icon/color pickers, habit history.
- Keep `TaskBottomSheet(...)` / `HabitBottomSheet(...)` public signatures and `TasksViewModel`/`RoutinesViewModel` event contracts stable so callers (`TasksScreen.kt`, `RoutinesScreen.kt`) need no changes.
- Improve testability: shared mechanics get one shared test suite instead of two parallel ~800–1300 line instrumented test files re-testing the same behavior.

**Non-Goals:**
- No visual/UX redesign of either editor.
- No new fields, no new capability (icons/colors are not added to tasks; periodicity is not added to habits).
- No change to `CreateTaskUseCase`/`UpdateTaskUseCase`/`CreateHabitUseCase`/`UpdateHabitUseCase`/`CreateOrUpdateHabitChainUseCase` result/error shapes as observed by callers — only their internal validation dispatch is consolidated.
- No database/schema changes.
- Not a general-purpose "form framework" for future unrelated editors (e.g. Settings, Categories) — scope is limited to what Tasks and Routines already share today, per the issue's explicit non-goal.

## Decisions

**1. Shared scaffold lives in `core/ui/editor/` (new package), not in either feature.**
Both `features/tasks` and `features/routines` already depend on `core/ui` for `TempoModalBottomSheet`, `ExpressiveChip`, etc. Putting the new shared pieces in `core/ui/editor/` keeps the dependency direction consistent (features depend on core, not on each other) and avoids a feature-to-feature dependency. Alternative considered: put it in `core/ui/components/` alongside existing components — rejected in favor of a dedicated `editor/` subpackage so the cohesive editor-scaffold API surface (snapshot diffing, autosave, footer, property row) is discoverable as a unit rather than mixed into the general component grab-bag.

**2. Snapshot/dirty-tracking/autosave becomes a generic, type-parameterized composable/state holder, not a shared data class.**
Each feature's snapshot shape differs (`TaskFormSnapshot` vs. `HabitFormSnapshot`/`ChainFormSnapshot`). Rather than force one snapshot data class on both, extract the *algorithm* — capture baseline on open, diff current vs. baseline for `hasUnsavedChanges`, debounce-dispatch autosave when the live snapshot differs from the last autosaved one, flush on dismiss if dirty and title non-blank — as a generic composable/remembered-state holder parameterized over a snapshot type `T` with an equality-based diff (`rememberEditorAutosaveState<T>(...)` or similar), reusing the existing `DebouncedSnapshotEffect` underneath. Each feature keeps its own snapshot data class and supplies it to the shared holder. Alternative considered: unify `TaskFormSnapshot`/`HabitFormSnapshot` into one shared snapshot type — rejected because the fields are genuinely different per feature (periodicity vs. icon/color/repeat-days) and forcing a union type would reintroduce exactly the "awkward abstraction" the issue explicitly warns against.

**3. Footer and property-row are plain shared composables taking explicit parameters (labels, enabled state, callbacks), not feature-aware.**
`TaskBottomSheetFooterSection` / `HabitBottomSheetFooterSection` differ only in the label/enabled-condition logic already computed by each feature's content composable — the visual/animation code is identical. The shared footer composable takes already-computed `confirmEnabled: Boolean`, `confirmLabel: String`, `onConfirm/onCancel/onDelete` etc., with no knowledge of tasks or habits. Same approach for the property-row layout (icon box + content slot). This keeps the shared components free of feature-specific branching, satisfying the issue's "extract reusable patterns without forcing unrelated domains into an awkward abstraction."

**4. Title/description validation is consolidated into a single shared domain helper, called from each use case, keeping each use case's own `Result`/error-mapping wrapper.**
Introduce a shared `EditableTextValidation` (or equivalent) helper in `core/domain/util/` (alongside existing `ValidationUtils`) that returns a common validation outcome (e.g. reusing/extending `ValidationResult`) for title/description. Each use case (`CreateTaskUseCase`, `UpdateTaskUseCase`, `CreateHabitUseCase`, `UpdateHabitUseCase`, `CreateOrUpdateHabitChainUseCase`) calls this helper and maps the outcome into its own existing `ValidationError`/`Result` type, so external result shapes are unchanged. The duplicated `ValidationErrorType` enums (`TITLE_EMPTY`, `TITLE_TOO_LONG`, `DESCRIPTION_TOO_LONG`) collapse into one shared enum reused by both features' error types (or the enum itself is shared directly if both features' error sealed classes can wrap it without changing their public shape). Alternative considered: leave use-case-level validation duplicated and only dedupe the ViewModel-level pre-check — rejected because the use-case-level duplication (identical `when` dispatch, identical enum members) is the larger and more error-prone duplication, and is fully internal so consolidating it carries no external-behavior risk.

**5. Editor-session-key derivation becomes a tiny shared helper function, not a composable.**
Both features derive a `rememberSaveable` key from `(editorSessionId, editingTargetId[, extra fields])`. A plain function `editorSaveableKey(sessionId: Long, vararg extra: Any?)` (or similar) is extracted; each feature passes its own extra discriminators (habits pass `selectedTab` in addition). No behavior change, just removes the duplicated construction.

**6. Shared UI test utilities are added incrementally, not a full rewrite of existing instrumented tests.**
Rather than rewriting `TaskBottomSheetTest.kt` / `HabitBottomSheetTest.kt` wholesale, add shared test-tag constants and/or small Compose test helpers for the newly-shared components (footer, property row) under a shared test-util location, and migrate the parts of each test file that assert on now-shared behavior (dirty-tracking, autosave flush, discard dialog, footer button states) to use them. Feature-specific assertions (periodicity chips, icon/color pickers, chain selection) stay in their respective files. This keeps the change reviewable and avoids a large, risky test-file rewrite as a side effect of a behavior-preserving refactor.

## Risks / Trade-offs

- **[Risk]** Generalizing the snapshot/autosave algorithm could subtly change autosave timing or discard-dialog suppression behavior (e.g. habits' extra `autoSelectedColor`/`autoSelectedIcon` dirty-suppression logic) if the generic holder doesn't accommodate feature-specific dirty-check overrides. → **Mitigation:** design the shared holder's diff function as a caller-supplied `(T, T) -> Boolean` (or equivalent), so feature-specific suppression logic (e.g. ignoring auto-selected icon/color) is expressed in the feature's own equality/diff function passed into the shared holder, not hardcoded into the shared code.
- **[Risk]** Consolidating validation enums/dispatch touches every create/update use case for both features simultaneously — a mistake affects both Tasks and Routines saving at once. → **Mitigation:** keep the change behind existing unit test suites (`CreateTaskUseCaseTest`, `UpdateTaskUseCaseTest`, `CreateHabitUseCaseTest`, `UpdateHabitUseCaseTest`, `CreateOrUpdateHabitChainUseCaseTest`) unmodified in their assertions (same boundary conditions, same expected `Result`/error values) so any regression in shared validation surfaces immediately; land use-case validation consolidation as its own task/commit separate from the UI scaffold extraction.
- **[Risk]** Moving shared composables into a new `core/ui/editor/` package could be mistaken for a place to add unrelated future editors (Settings, Categories), reintroducing the "awkward abstraction" the issue warns against. → **Mitigation:** scope the extracted API surface strictly to what Tasks and Routines currently share (documented in this design's Decisions); do not generalize parameters beyond what both current call sites need.
- **[Trade-off]** This refactor intentionally does not touch `CategoryEditSheetTest.kt` / the category edit sheet, even though it follows a lighter version of the same pattern (noted during investigation) — including it would expand scope beyond the issue's two named domains (tasks, habits/routines). Left as a natural follow-up, not part of this change.

## Migration Plan

Pure internal refactor with no data migration, no feature flag, and no rollout sequencing beyond normal code review — behavior is expected to be bit-for-bit unchanged for end users. Rollback is a normal git revert if instrumented/unit tests or manual verification (`TaskBottomSheetTest`, `HabitBottomSheetTest`, `TasksViewModelTest`, `RoutinesViewModelTest`, plus manual smoke test of create/edit/autosave/discard/delete on both a task and a habit) reveal a regression. Land in stages (tasks.md sequences this): (1) shared constants + footer + property row + editor-key helper, (2) shared snapshot/autosave scaffold, (3) shared validation consolidation, (4) shared test utilities — each stage independently buildable/testable so a problem in a later stage doesn't block reverting just that stage.

## Open Questions

- Exact naming for the shared package/API (`core/ui/editor/` vs. `core/ui/components/editor/`) and for the generic autosave-state holder — left to implementation, does not affect the design's substance.
- Whether the consolidated `ValidationErrorType` enum should be shared directly by both features' error sealed classes or kept as an internal detail of the new shared validation helper with each feature's existing enum mapped from it — left as an implementation-time call in tasks.md, guided by whichever keeps use-case public result shapes most obviously unchanged.
